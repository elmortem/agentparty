# AgentParty — Review

Результаты ревью кода и обсуждения ожиданий от библиотеки. Документ служит отправной точкой: по каждому пункту со статусом "К ТДД" планируется отдельный technical design, прежде чем что-то менять в коде.

**Контекст.** AgentParty — библиотека транспорта и протокола общения для личных LLM-агентов. Не энтерпрайз. Цель — простота и минимум сущностей. Оркестрация, сессии, агент-логика, ACL пользователей — за пределами библиотеки.

---

## 1. Критические баги и thread-safety

### 1.1. Thread-safety Router
**Статус:** К ТДД.

`Router._servers`, `_routingTable`, `_handlers`, `_feedHandlers` — обычные `List`/`Dictionary`. При параллельных входящих от разных серверов (FileServer через пул потоков, TelegramServer через update handler) запись в `_routingTable[message.ClientId] = server` гоняется без лока. `Register`/`Unregister` во время работы тоже не потокобезопасны.

**Направление:** `ConcurrentDictionary` для routing table, lock для списков и handlers-коллекций.

### 1.2. `GetAwaiter().GetResult()` внутри event handler'а Router
**Статус:** К ТДД.

В `Router.Register` обработчик `MessageReceived` при блокировке команды синхронно вызывает `server.SendAsync(...).GetAwaiter().GetResult()`. Риски: deadlock на контекстах с SynchronizationContext, блокировка транспортного потока, необработанные исключения из подписчиков ломают polling loop.

**Направление:** event handler — `async void` или сделать обработку команд через очередь/канал. Добавить try/catch вокруг `Invoke` подписчиков.

### 1.3. TelegramServer.StopAsync не ждёт окончания polling
**Статус:** К ТДД.

`StartReceiving` крутится в фоне, `StopAsync` дергает cancel и возвращает `Task.CompletedTask`. После этого `HandleUpdateAsync` ещё может выполниться и поднять `MessageReceived` уже после `_isRunning = false`.

**Направление:** сохранять Task от receiver'а (через свою реализацию `IUpdateHandler`) и ждать его в Stop.

### 1.4. Утечка `_sentMessageMap` в TelegramServer
**Статус:** К ТДД.

`ConcurrentDictionary<int, string>` растёт при каждой отправке Choice/List и никогда не чистится. Долгоживущий бот → утечка памяти. Дополнительно: при рестарте процесса mapping теряется, и callback_query по старой кнопке даст `Response.To = telegramMsgId`, который агент не сможет сопоставить — silent break.

**Направление:** TTL/LRU для ограничения размера. Для устойчивости к рестарту — рассмотреть упаковку `message.Id` прямо в `callback_data` (префикс/разделитель), чтобы не нуждаться в in-memory mapping.

### 1.5. FileServer — возможная двойная доставка и отсутствие claim
**Статус:** К ТДД.

Пайплайн: `ReadAllText` → `Delete` → `Invoke`. Между чтением и удалением процесс может быть убит — при рестарте файл будет обработан снова. Watcher и polling могут параллельно войти в `ProcessIncomingFiles` для одного файла.

**Направление:** атомарный claim через `File.Move` в подкаталог `processing/` (или суффикс по PID), обработка, удаление. Синхронизация между watcher и polling.

### 1.6. FileServer — `.tmp` файлы не чистятся
**Статус:** Простой фикс, в ТДД можно включить вместе с 1.5.

Writer упал между `WriteAllText` и `Move` → `.tmp` остаётся навсегда.

**Направление:** на старте сносить все `.tmp` в директориях сервера/клиента.

### 1.7. FileClient — multi-client коллекционирование в общем outgoing/
**Статус:** К ТДД.

Сейчас все клиенты читают общий `outgoing/`, фильтруют по своему `ClientId`. Каждый polling перечитывает все файлы, включая чужие. Для большого числа клиентов — квадратичная нагрузка.

**Решение (согласовано):** завести подкаталоги `outgoing/{clientId}/`. FileServer пишет прямо в нужный подкаталог, FileClient смотрит только в свой. Чистка чужих файлов — не задача клиента; если они лежат — это проблема того, кто не подключился.

### 1.8. Отсутствует try/catch вокруг event invoke
**Статус:** К ТДД (объединить с 1.2).

Если подписчик на `MessageReceived`/`FeedReceived` кинет исключение, оно прилетит обратно в транспорт и уронит polling loop (FileServer) или update handler (Telegram).

**Направление:** оборачивать `Invoke` в try/catch, логировать через `IRawLogger`.

## 2. Несоответствия README/TDD и кода

### 2.1. Markdown в Telegram
**Статус:** Согласовано, фиксим.

README и TDD обещают Markdown через `SendMessage`. В коде `SendMessage` без `parseMode` — плетит plain text. Info-list же идёт как `ParseMode.Html` — несогласованно.

**Решение:** для `text`, `choice`, action-items в `list` — `ParseMode.Markdown` (или MarkdownV2, определимся в ТДД). Info-list тоже привести к Markdown.

### 2.2. Разные форматы Timestamp
**Статус:** К ТДД.

`Message.Timestamp` — ISO-строка (дефолт System.Text.Json). `FeedMessage.Timestamp` — Unix seconds через `UnixDateTimeConverter`. По одной файловой шине летят разные форматы.

**Направление:** выбрать один формат. Unix seconds удобнее для межязыковой совместимости, ISO — для читаемости логов. Обсудим.

## 3. Ограничения дизайна протокола

### 3.1. Приватное vs публичное общение в IMessage
**Статус:** К ТДД (на будущее).

Базовая модель — приватное 1:1 общение. Фид — отдельный абстрактный поток без привязки к юзеру. Группа в тупик не упирается только потому, что сейчас всё падает в feed.

**Решение (согласовано):** оставить как есть. Если когда-то понадобится активный агент в группе/треде — добавить флаг "private/public" в `IMessage` или отдельный тип. Без изменений до реального use case.

### 3.2. ClientId = chatId в Telegram
**Статус:** Обсуждалось, остаётся как есть.

Для приватного общения это нормально — chatId уникален. Для групп — тема 3.1.

### 3.3. Метаданные (`Headers`) в IMessage
**Статус:** Отклонено.

Предлагался `IDictionary<string,string>? Headers` для correlation-id, traceparent и т.п. Решено не добавлять — не усложняем схему. Библиотека для личных агентов, не энтерпрайз-протокол.

### 3.4. Тег сервера в сообщениях
**Статус:** Отклонено для core.

Полезно было бы знать, с какого сервера пришло сообщение (например, для группировки в логах агента). Решено: это уровень агента — под конкретные нужды можно написать свой роутер, наследник или обёртка. В библиотеку не тащим.

### 3.5. Версионирование envelope
**Статус:** Отложено.

Нет `version`/`schema` в конверте. Breaking changes в Content-моделях тихо сломают старые клиенты. Для личных агентов пока не приоритет — при реальном изменении формата добавим.

### 3.6. Media / attachments
**Статус:** Отложено, зарубка.

Только текст. Для базовых сценариев (личные агенты, ПМ, дайджесты) не критично. Если появится реальный use case (не развлекательный) — отдельное ТДД.

### 3.7. Streaming / editMessageText
**Статус:** Отклонено.

Стриминг и редактирование — специфика Telegram. Другие транспорты (файл, REST, WebSocket, inter-agent) это не поддерживают или поддерживают по-разному. Наталкивать это в единую абстракцию — переусложнить. Если в логе хранится "отредактированное" сообщение — непонятно, что делать со старым состоянием.

**Решение:** оставить атомарные сообщения. "Thinking"-нотификация покрывает UX "я думаю".

## 4. Транспортные улучшения

### 4.1. Rate-limit, back-pressure и обработка 429 в Telegram
**Статус:** К ТДД.

Telegram лимит: ~30 msg/sec глобально, 1 msg/sec на чат, 20 msg/min в группу. При превышении HTTP 429 с Retry-After. Сейчас `TelegramRenderer.SendAsync` валится первым 429 exception'ом.

**Решение (согласовано):** держать всё на стороне библиотеки, чтобы агент не думал про лимиты и ретраи.
- Очередь отправки в памяти (без persistence на диск — процесс упал, очередь потеряна, это приемлемо).
- Rate-limit bucket per-chat + global (`SemaphoreSlim` или `System.Threading.RateLimiting`).
- Перехват `ApiRequestException` с code 429 → ждать `Retry-After` → повторить. Ограниченное число попыток.
- Прочие retryable ошибки (timeout, network) — экспоненциальный backoff, ограниченно.
- Контракт `SendAsync`: возвращается, когда сообщение либо успешно отправлено, либо исчерпаны попытки (exception). На время ожидания — просто задерживает вызов (конкретика — в ТДД).

### 4.2. CancellationToken не пробрасывается в File*
**Статус:** Мелочь, включим в ТДД по 1.5.

`WriteAllText`/`ReadAllText`/`Delete` — синхронные без ct. Для локального диска не критично, но формально контракт нарушается.

### 4.3. `long.Parse` без `InvariantCulture`
**Статус:** Простой фикс.

В `FeedSource.Parse` и в `TelegramServer.SendAsync` — `long.Parse(s)`. Добавить `CultureInfo.InvariantCulture` везде, где парсятся ID.

### 4.4. `long.Parse(clientId)` в Telegram.SendAsync — FormatException
**Статус:** К обсуждению.

Если Router ошибся маршрутом (коллизия ClientId между транспортами) — `FormatException` вместо понятной ошибки. Варианты: ловить и кидать свою, либо полагаться на то, что коллизий не бывает.

## 5. Инфраструктура

### 5.1. Telegram.Bot в core-пакете
**Статус:** К ТДД (после стабилизации).

Сейчас все, кто юзает FileServer, тянут `Telegram.Bot` и его зависимости. Напрашивается разделение: `AgentParty.Core`, `AgentParty.Telegram`, `AgentParty.File`, `AgentParty.Console`.

**Не делаем прямо сейчас** — сначала стабилизируем core. Разделение — отдельным ТДД перед 1.0.

### 5.2. ILogger / observability
**Статус:** К обсуждению.

`IRawLogger` есть, но примитивный. Нужен ли полноценный `Microsoft.Extensions.Logging.ILogger`? Для личных агентов, возможно, хватит расширить `IRawLogger` уровнями (Info/Warn/Error). Решаем в отдельном ТДД.

### 5.3. IAsyncDisposable
**Статус:** Простой фикс.

`Dispose` сейчас блокирует через `.GetAwaiter().GetResult()`. Добавить `IAsyncDisposable` параллельно с `IDisposable`.

### 5.4. `Router.AllowedCommands` — информационный геттер
**Статус:** К обсуждению.

Возвращает union всех серверов, но реальная фильтрация идёт по серверу-источнику. Можно ввести в заблуждение.

**Направление:** либо переименовать (например, `EffectiveAllowedCommands`), либо убрать из публичного API Router и оставить только у серверов.

### 5.5. DI extensions
**Статус:** Отложено.

Не критично для текущего use case. Когда появится проект на ASP.NET Core — добавим `services.AddAgentParty()...`.

## 6. Что НЕ делаем на уровне библиотеки

Зафиксируем явно, чтобы не возвращаться:

- **Proactive-сообщения от агента, восстановление clientId после рестарта.** Уровень агента. AgentParty — доставка + протокол. Агент сам хранит свои clientId'ы где ему удобно.
- **ACL per-user.** Whitelist команд в AgentParty — грубая транспортная фильтрация. Тонкая авторизация — уровень агента.
- **Persistence сообщений, retry, delivery guarantees.** Личные агенты — at-most-once приемлем. Если понадобится at-least-once — через внешний брокер, не через AgentParty.
- **Streaming, editMessageText, сообщения с эволюцией.** См. 3.7.
- **Сессии, контексты, диалоги.** Уровень агента.

## 7. Приоритизация ТДД

Порядок, в котором предлагаю разбирать каждый пункт отдельным ТДД:

1. **Thread-safety + event handlers Router** (1.1, 1.2, 1.8) — один ТДД.
2. **TelegramServer lifecycle + sentMessageMap** (1.3, 1.4) — один ТДД.
3. **FileServer atomic claim + .tmp cleanup + подкаталоги outgoing** (1.5, 1.6, 1.7, 4.2) — один ТДД.
4. **Markdown в Telegram + rate-limit** (2.1, 4.1) — один ТДД.
5. **Timestamp формат** (2.2) — один ТДД или вместе с п.3.
6. **Мелочи без ТДД:** `InvariantCulture` (4.3), `IAsyncDisposable` (5.3), `FormatException` в SendAsync (4.4) — можно фиксить по мере работы с основными ТДД.

На будущее (после стабилизации core): разделение пакетов (5.1), logger (5.2), версионирование (3.5), media (3.6), публичный/приватный флаг (3.1).
